1) В файле example 1.jpg изображена Star Schema с транзакционной таблицей фактов для первого пункта задания. В качестве бизнес-ключей в таблицах измерений я выбрала следующие:
Inventory_Num (Инвентарный номер присваивается каждому инвентарному объекту для ведения аналитического учета внутри организации, закрепляется за объектом на все время его эксплуатации. Может поменяться, если автосалон переедет в новое здание) для Car_Showrooms; Registration_Num (Регистрационный номер для автомобиля при отпуске его со склада, может поменяться при перемещении со склада на склад) для Auto; Personnel_Num (Табельный номер работника, в некоторых организациях он может поменяться в случае увольнения и приема на работу через определенный срок) для Managers; Passport_Num (Номер паспорта) для Clients.
Durable keys: VIN_Code(Уникальный серийный номер, используется для идентификации автомобилей по всему миру) для Auto; Identification_Num (Идентификационный номер пасспорта) для Managers и Clients.
2) В файле example 2.jpg изображена Star Schema с таблицей фактов по типу Periodic Snapshot для второго пункта задания. Рассматривается загрузка для каждого отдельного склада в конце дня. Одной из метрик в таблице фактов я сделала Num_of_Unpopular_Cars - число непопулярных машин на складе к концу дня. Предполагается, что на входе в склад мы проверяем, популярная ли машина, например по атрибуту isPopular у Auto, что в свою очередь устанавливается каким-либо условием из динамики продаж в первом пункте (например, за неделю иди декаду до начала этого дня). В качестве бизнес-ключа выбран Inventory_Num (инвентарный номер складского помещения) для таблицы измерений Stores. 
3) В файле example 3.jpg изображена Star Schema с таблицей фактов по типу Accumulated Snapshot для третьего пункта задания. Предполагается, что все этапы проходят посредством некой интернет-площадки, поэтому информации о менеджерах, как при продаже непосредственно в автосалоне, здесь нет. Для дат и времени прохождения этапов я использовала Role-Play Dimensions. В качестве business keys таблицах измерений были взяты Registration_Num для Auto; Tax_Identifiction_Num для Manufacturers(например, в РБ это единый уникальный номер, присваиваемый каждому плательщику при постановке на учёт в налоговом органе, - учетный номер плательщика или УНП. Может поменяться при некоторых видах реорганизации, таких как разделение, слияние, выделение); Inventory_Num для Stores; Passport_Num для Clients.
Durable keys: VIN_Code для Auto, Coordinates для Location, Identification_Num для Clients.
